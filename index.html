01 / 05 2025년 9월 ~ 2026년 2월

사내 문서를 근거로 답하는 챗봇

처음부터 사내 문서를 근거로 답하는 RAG 챗봇으로 시작했습니다. 카테고리별로 문서를 올려 두면 그 안에서 찾아 답하고, 어떤 문장을 근거로 삼았는지 출처를 함께 보여 줍니다.

문서 등록이 관리자를 거치면 지식이 잘 쌓이지 않는다고 봤습니다. 그래서 업로드하면 파싱과 임베딩, 저장까지 자동으로 처리되는 파이프라인을 만들어 사용자가 직접 지식을 쌓게 했고, 모델도 하나에 묶지 않고 OpenAI와 Anthropic, Gemini를 골라 쓰도록 했습니다.

사내 지식 검색은 어떤 문서에서 나온 답인지 출처와 함께 보여 줍니다.

02 / 05 2026년 3월 ~ 4월

챗봇을 넘어 에이전트로

문서를 검색해 답하는 것만으로는 부족한 질문이 있었습니다. 누가 이번 달에 얼마를 썼는지 같은 건 문서가 아니라 업무 데이터를 봐야 했습니다.

그래서 웹 검색과 업무 데이터 조회를 도구로 붙이고, 검색을 한 번에 끝내지 않고 결과가 부족하면 범위를 넓혀 다시 찾게 했습니다. 기능을 늘리는 만큼 운영도 받쳐 줘야 해서, 이 시기에 관리자 대시보드로 사용량과 비용을 들여다보고 사내 계정 로그인과 피드백 창구를 함께 갖췄습니다.

웹 검색 결과는 출처와 함께 답변에 표시됩니다.

관리자 대시보드에서 사용량과 모델별 비용을 봅니다.

03 / 05 2026년 5월

프로젝트와 문서 생성

파일과 지침을 묶어 프로젝트를 만들면 그 안에서만 근거를 찾아 답하게 했습니다. 발표 자료가 필요하면 pptx를 만들어 채팅에 첨부하는 문서 생성도 이때 붙였습니다.

오래 쓰는 서비스라 안정성도 함께 손봤습니다. 대화가 길어지면 토큰 예산 안에서 요약해 기억하고, 끊긴 응답은 다시 접속해도 이어 보이고, 반복되는 프롬프트는 캐싱하고, 모델마다 한도와 비용을 걸어 관리했습니다.

프로젝트는 파일과 지침을 묶어, 그 안에서만 근거를 찾아 답합니다.

생성한 문서는 채팅 옆 뷰어에서 슬라이드를 페이지째 미리봅니다.

04 / 05 2026년 6월

코드와 아티팩트, 그리고 단일 에이전트

표를 눈대중으로 요약하는 대신, 계산이 필요한 요청은 코드를 작성해 격리된 샌드박스에서 실행하도록 했습니다. HTML을 만들어 바로 렌더해 보여 주는 아티팩트와, 코드로 문서를 찍어내는 생성 경로도 이때 품질을 끌어올렸습니다.

더 큰 변화는 내부 구조였습니다. 처음에는 Supervisor 노드가 질문을 분류해 경로를 정했지만, 분류 노드를 걷어내고 에이전트가 스스로 도구를 고르는 단일 ReAct 구조로 옮겼습니다. 진입 경로만 결정론으로 정하고, 그 뒤 도구 선택은 에이전트가 판단합니다.

질문이 답변이 되기까지. 경로 결정, 도구 실행, 과정 표시, 답변으로 이어집니다.

계산이나 시각화 요청은 코드를 실행해 인터랙티브 차트로 보여 줍니다.

05 / 05 2026년 7월

구조 정리와 사내 지식 개방

옮기기 시작한 구조를 마무리한 달입니다. 쓰지 않게 된 분류 노드와 죽은 경로를 걷어내고, 사내 지식 검색도 도구로 바꿔 하나의 에이전트 안에 넣었습니다. 최신 모델을 기본으로 올리고, 웹페이지 생성 결과의 결도 다듬었습니다.

그리고 그동안 일부만 쓰던 사내 지식 챗봇을 전 직원에게 열었습니다.

부서와 업무별로 만들어 둔 사내 지식 챗봇 목록입니다.
