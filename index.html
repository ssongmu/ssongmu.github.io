사내 지식 챗봇의 탄생

처음부터 사내 문서를 근거로 답하는 RAG 챗봇으로 시작했습니다. 카테고리별로 문서를 올려 두면 그 안에서 찾아 답하고, 어떤 문장을 근거로 삼았는지 오른쪽 패널에서 원문까지 열어 볼 수 있게 했습니다.

형태는 일반적인 챗봇이지만, 답하기 전에 사내 문서를 먼저 검색했습니다. 답변 스트리밍을 붙이고, 모델도 하나에 묶지 않고 Claude와 Gemini, OpenAI를 골라 쓰도록 했으며, 이미지를 붙여 묻는 것도 이때 넣었습니다.

부서와 업무별로 만들어 둔 사내 지식 챗봇 목록입니다.

02 / 05 2026년 3월 ~ 4월

챗봇을 넘어 에이전트로

문서를 검색해 답하는 것만으로는 부족한 질문들이 있었습니다. 누가 이번 달에 얼마를 썼는지 같은 건 문서가 아니라 데이터베이스를 봐야 했습니다.

그래서 검색을 한 번에 끝내지 않고 스스로 다시 찾게 하는 retrieval, 질문을 SQL로 옮기는 에이전트, 바깥을 뒤지는 웹 검색을 그래프로 나눠 붙였습니다. 기능을 늘리는 만큼 운영도 받쳐 줘야 해서 이 시기에 관리자 대시보드로 사용량과 비용을 들여다보고, 사내 계정으로 로그인하고, 무거운 업로드를 큐로 돌리고, 사용자 피드백을 받는 축을 같이 깔았습니다.

웹을 검색해 답하고, 문장 끝에 근거가 된 출처를 붙입니다.

관리자 대시보드에서 부서별 사용량과 모델별 비용을 봅니다. 최근 14일 5,520건, 프롬프트 캐싱으로 비용을 37% 줄였습니다.

03 / 05 2026년 5월

프로젝트와 문서 생성

파일 몇 개와 지침을 묶어 프로젝트를 만들면 그 안에서만 근거를 찾아 답하게 했습니다. 발표 자료가 필요하면 pptx를 만들어 주는 생성 엔진도 이때 처음 붙였는데, 레이아웃이 어그러지면 스스로 한 번 검사해 고치도록 했습니다.

오래 쓰는 서비스라 안정성도 함께 손봤습니다. 대화가 길어지면 토큰 예산 안에서 요약해 기억하고, 끊긴 응답은 다시 접속해도 이어 보이고, 반복되는 프롬프트는 캐싱하고, 모델마다 한도와 비용을 걸어 관리했습니다.

발표 자료를 만들어 달라고 하면 슬라이드를 짜서, 내려받을 파일로 돌려줍니다.

04 / 05 2026년 6월

코드와 아티팩트, 그리고 단일 에이전트

가장 많이 갈아엎은 달입니다. 표를 눈대중으로 요약하던 걸 그만두고, 코드를 실제로 돌려서 정확히 집계하게 했습니다. HTML을 만들어 바로 렌더해 보여 주는 아티팩트, pptx와 docx, xlsx, pdf를 코드로 찍어내는 문서 생성도 이때 레퍼런스를 옆에 두고 품질을 끌어올렸습니다.

더 큰 변화는 내부 구조였습니다. 그동안은 supervisor가 여러 전문 에이전트에게 일을 나눠 주는 구조였는데, 이걸 도구를 스스로 골라 쓰는 하나의 ReAct 에이전트로 옮기기 시작했습니다. 판단하는 자리를 한 곳으로 모으니 흐름이 단순해지고 어디서 무슨 결정이 났는지 쫓아가기 쉬워졌습니다.

supervisor가 나눠 주던 일을, 도구를 스스로 고르는 하나의 에이전트로 옮겼습니다.

요청을 받으면 슬라이드 작성과 문서 생성 단계를 보여 주고, pptx 파일로 완성합니다.

05 / 05 2026년 7월

하나로 수렴, 전 직원에게

옮기기 시작한 걸 마무리한 달입니다. 쓰지 않게 된 supervisor와 죽은 노드를 걷어내고, 마지막까지 따로 놀던 사내 지식 검색마저 도구로 바꿔 하나의 에이전트 안으로 넣었습니다. 최신 모델을 기본으로 올리고, 웹페이지를 만들 때 나오는 결과물의 결도 다듬었습니다.

그리고 그동안 일부만 쓰던 사내 지식 챗봇을 전 직원에게 열었습니다.
